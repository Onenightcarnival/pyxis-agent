# 013：MCP —— 把一个 MCP server 变成一批 Tool 子类

## 目的

MCP 就是"一批远端 JSON-RPC 工具"，和 pyxis `Tool` 在形态上严格同构
（name + inputSchema + call vs. class + fields + `run()`）。提供一个
最小适配器：声明一个 `MCPServer`（Pydantic 数据）、`async with
mcp_toolset(server)` 拿到 `list[type[Tool]]`、塞进判别式联合，剩下的
就是普通 pyxis agent loop。

**不为 MCP 扩 `Tool` 契约**：`Tool.run()` 仍是同步 `-> str`；传输层
（HTTP / stdio）的复杂度全部吸收在 adapter 里，不外泄。

## API 草图

```python
from pyxis import Tool, flow, step
from pyxis.mcp import MCPServer, StdioMCP, HttpMCP, mcp_toolset

fs = MCPServer(name="fs", transport=StdioMCP(
    command="uvx", args=["mcp-server-filesystem", "/tmp"]))
web = MCPServer(
    name="web",
    transport=HttpMCP(url="https://example.com/mcp", headers={"Authorization": "..."}),
    include=["fetch_url"],  # 精确名白名单
)

@flow
async def agent(q: str) -> str:
    async with mcp_toolset(fs) as fs_tools, mcp_toolset(web) as web_tools:
        tool_classes = [Calculate, Finish, *fs_tools, *web_tools]
        Action = Annotated[tuple(tool_classes), Field(discriminator="kind")]
        # ... 拼 Decision，跑 agent loop，await d.action.run() 照常
```

## 验收标准

- `MCPServer(BaseModel)` 字段：
  - `name: str`（人类标识，碰撞处理见下）；
  - `transport: StdioMCP | HttpMCP`（判别式联合，`kind: Literal[...]`）；
  - `include: list[str] | None = None`（精确名白名单，`None` 表示全收）；
  - `exclude: list[str] = []`（精确名黑名单，`include` 之后再过滤）；
  - `timeout_s: float = 30.0`（单次 `tools/call` 的超时）。
- `StdioMCP` 字段：`kind: Literal["stdio"] = "stdio"`、`command: str`、
  `args: list[str] = []`、`env: dict[str, str] = {}`。
- `HttpMCP` 字段：`kind: Literal["http"] = "http"`、`url: str`、
  `headers: dict[str, str] = {}`。
- `mcp_toolset(server) -> AsyncContextManager[list[type[Tool]]]`：
  - 入口建立连接 → 发 `initialize` → `tools/list` → 按 `include`/`exclude`
    过滤 → 为每个工具生成 `Tool` 子类；
  - 工具类名：`snake_case(tool.name) -> PascalCase`；
    `kind: Literal[tool.name] = tool.name`；
  - 字段由 `inputSchema`（JSON Schema）翻译而来：支持
    `str / int / float / bool / list / dict / Optional`；
    `required` 里的为必选，否则默认 `None`；
    不支持的类型抛 `TypeError` 并点名哪个字段哪个关键字。
  - `run(self) -> str`：同步调 `tools/call`，把 `result.content` 里
    `type == "text"` 的 chunk 用换行拼接；非文本内容在首版忽略。
  - 名字碰撞（同一 `mcp_toolset` 作用域内、跨多个 `mcp_toolset` 的 `*`
    展开都算）：由用户在 agent loop 里拼 list 时负责检测；adapter 本身
    只在**单个 server 内部**的重名（MCP 协议本不允许）抛 `ValueError`。
  - 退出时关连接：子进程 `terminate + wait`；HTTP `Client.close()`。
- 传输实现：
  - `HttpMCP`：`httpx.Client` 发 `POST {url}`，body 是单条 JSON-RPC
    请求。**对齐 MCP 2024-11-05 Streamable HTTP 规范**：
    - 请求头始终带 `Accept: application/json, text/event-stream`；
    - 响应按 `Content-Type` 分支解析——`application/json` 直接 `.json()`，
      `text/event-stream` 按 SSE 事件块解析（抓 id 匹配的 `data:` payload）；
    - `Mcp-Session-Id` 响应头会被记录，后续请求作为请求头回写；
    - `initialize` 成功后自动发 `notifications/initialized` 通知（FastMCP
      等严格实现要求这条，否则拒绝后续请求）。
    这意味着 `HttpMCP` 可以直接对接 FastMCP 写的 server。
  - `StdioMCP`：`subprocess.Popen` 持久子进程；`stdin` 写一行 JSON，
    `stdout` 按行读，`id → response` 字典关联，`threading.Lock` 串行
    化；`initialize` 在入口做一次。
- `trace()` 自动覆盖（adapter 生成的 `run()` 走普通同步路径，`Step` 的
  记录链不需要任何额外集成——**这条是 C1 决策的验收**）。

## 不做

- 不扩 `Tool` 契约（不加 `arun`、不分裂 `AsyncTool`）。
- 不做**老 SSE 传输**（`GET /sse` 长连接 + `POST /messages` 两根管子那种，
  已被 Streamable HTTP 取代）。注意：这跟 Streamable HTTP 的 **SSE 响应
  体**不是一回事——后者是同一次 POST 里 server 选择的 Content-Type，
  `HttpMCP` **已完整支持**。
- 不做 MCP resources / prompts / sampling / roots（首版聚焦 `tools`）。
- 不做全局 MCP registry、不做连接池、不做懒连接。生命周期绑 `async with`。
- 不做 stdio / HTTP 的并发调用。当前 agent loop 是串行的，`Lock` 是串
  行化而非并发支持；想并发以后再开 spec。
- 不做断线重连 / 自动重启子进程。`async with` 作用域结束就关。
- 不做 tool schema 的运行时动态刷新（`tools/list` 只在入口调一次）。
- 不做 `ToolSet` 抽象 protocol。只有 MCP 一家实现时，抽象就是空话；
  将来若出现第二个 source（OpenAPI 等），再提取。
- 不做 tool 名自动加前缀。碰撞由使用者显式解决（改 `include`、包一层
  `@tool` 重命名、或不同 flow 作用域隔离）。
