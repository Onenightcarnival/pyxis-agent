# 009：错误可见性 —— 把失败写进 trace

## 目的

结构化输出的 agent 调试链条最痛的一环是：LLM 产出的 JSON 过不了 Pydantic
校验，instructor 耗尽重试后抛 `InstructorRetryException`，**但 trace 里
什么都没有**——用户只拿到一个异常，没有上下文可查。

让每一次失败也能被写进 trace：
- `TraceRecord.error: str | None` 填入失败原因；
- `output` 字段放宽为 `BaseModel | None`，失败时为 None；
- `Step` / `AsyncStep` 捕获异常写一条记录后再**原样重抛**，用户行为不变，
  但 trace 完整。

## API 草图

```python
from pyxis import FakeClient, Trace, step, trace

fake = FakeClient([])  # 空队列会让调用时抛 RuntimeError

@step(output=Plan, client=fake)
def plan(x: str) -> str:
    return x

with trace() as t:
    try:
        plan("x")
    except RuntimeError:
        pass

# 失败的调用仍然被记录
(rec,) = t.records
assert rec.error is not None
assert "耗尽" in rec.error
assert rec.output is None

# 聚合助手
assert len(t.errors()) == 1
```

## 验收标准

- `TraceRecord.error: str | None = None`；默认 `None`。
- `TraceRecord.output` 的类型从 `BaseModel` 放宽为 `BaseModel | None`。
- 在 `Step.__call__` / `AsyncStep.__call__` 里，当 `client.complete` /
  `client.acomplete` 抛出任意 `Exception` 时：
  1. 写一条 `TraceRecord(..., output=None, error=f"{类型}: {消息}")`；
  2. 用 `raise` 重抛原异常（保留 traceback，**不**封装）。
- `Trace.errors() -> list[TraceRecord]` 返回所有 `error is not None` 的记录。
- `TraceRecord.to_dict()` 正确处理 `output is None`（序列化为 `null`）
  与 `error`（总是包含该键，成功时值为 `null`）。
- 成功路径的历史测试全部不变（output 仍为 BaseModel 实例，error 为 None）。

## 不做（留给后续迭代）

- 暴露 `pydantic.ValidationError` 的结构化详情（目前把整个 `ValidationError`
  `repr` 进字符串即可满足调试）。
- 自动把 validation error 喂回 LLM——instructor 已在 `max_retries` 范围
  内做过了；耗尽后停。
- 在 Tool.run() 抛错时写 trace——Tool 是用户代码，调用它的循环也是用户
  代码，`trace()` 只捕 Step 调用。
