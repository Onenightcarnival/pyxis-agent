# 010：流式输出 —— 让思维链的字段被观察到逐个填完

## 目的

声明式思维链最有"可视化"张力的一点：LLM 按 schema 字段顺序自上而下
填字段，这个过程本身就是思维链在"被观察"。流式调用让用户在 LLM 还没
产出完整实例时就看到 `observation` 先出现，再看到 `reasoning`，最后才
是 `conclusion`——思维链"活"了。

底层直接借 instructor 的 `create_partial` 做 partial streaming；框架只
增加：
- `Step.stream(...)` → `Iterator[T]`、`AsyncStep.astream(...)` → `AsyncIterator[T]`；
- 流完整消费完时写一条 TraceRecord（`output` = 最后一帧，`usage = None`
  因为 partial 流里 instructor 不提供 usage）；
- 流中途任意异常时写一条 `error` 非空的 TraceRecord 再原样重抛。

## API

```python
@step(output=Plan)
def plan(x: str) -> str:
    """..."""
    return x

# 同步流式
for partial in plan.stream("hello"):
    print(partial)   # Plan 字段逐渐填满
# 迭代结束后最后一帧就是完整的 Plan

# 异步流式
@step(output=Plan)
async def aplan(x: str) -> str:
    """..."""
    return x

async for partial in aplan.astream("hello"):
    ...
```

## 验收标准

- `Client` 协议加 `stream[T](messages, response_model, model, *, max_retries=0) -> Iterator[T]`；
  `AsyncClient` 加 `astream[T](...) -> AsyncIterator[T]`。
- `FakeClient.stream` / `astream` 消耗一个 response 并 yield 一次（模拟
  "一帧流"）；`.calls` 仍记一次；队列耗尽抛 `RuntimeError`。
- `InstructorClient.stream` 走 `create_partial(...)`；`astream` 走异步版本。
- `Step.stream(*args, **kwargs) -> Iterator[T]`：
  - 构造 messages 与同步调用同构；
  - 透传 client.stream 的每一帧；
  - 流**完整结束**后写一条 TraceRecord（`output` = 最后一帧，`usage=None`）；
  - 流中途任意异常时写一条 error 记录再 `raise`。
- `AsyncStep.astream(*args, **kwargs) -> AsyncIterator[T]`：对偶行为，用
  `async for` 驱动。
- 用户在 `with trace():` 外调用 stream 仍然能用，只是 trace 不写入（与现有
  非流式 step 行为一致）。
